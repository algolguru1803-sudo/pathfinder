# ADR-0008: Свой ответ и коммент к варианту поверх существующего draft-контракта (0 правок сервера)

- **Дата:** 2026-06-13
- **Статус:** принято
- **Задача:** dashboard-feedback-enhancements

## Контекст

Нужны были две возможности обратной связи в дашборде: (1) человек может ответить на `choice`-вопрос
**своей формулировкой**, а не только выбрать опцию; (2) человек может **прокомментировать каждый
демо-вариант** явно, в т.ч. вариант без `caption` (через select-to-comment это было невозможно).

Бэкенд (`scripts/server.py`) **агностичен к содержимому items**: `/draft` мерджит item фиксированной
формы и хранит его, `/submit` пересылает весь draft в `submissions/<n>.json`, `replies.json` пишет
агент — сервер ничего не валидирует (`exploration-backend.md:11`). Но набор полей item в `_draft_add`
**жёстко фиксирован** (`scripts/server.py:720`): любое новое поле молча отбрасывается. И для `answer`
сервер держит **ровно один** item на `questionId` (дедуп `:730`).

## Решение

1. **Свой ответ = `answer` вне `options`.** Поле «Свой ответ» под radio-опциями шлёт обычный
   `POST /draft {kind:"answer", questionId, text}` (как и выбор опции). Никакого отдельного поля/флага
   «это свободный ответ». Различение — по факту: `answer.text`, не совпавший ни с одной `options`, и
   есть свободный ответ. На вопрос — **один answer** (серверный дедуп), поэтому свой ответ и выбор
   опции **взаимоисключающи**: ввод своего текста снимает radio, выбор опции чистит поле (клиентская
   часть в `wireBlocks` `:760`/`:766`).
2. **Коммент к варианту = `comment` с `blockId = variants[].id`, `selectedText:""`.** Тот же контракт,
   что у select-to-comment (`sendComment`), через тонкую обёртку `saveVariantComment` (`:898`). id
   варианта кладётся в **существующее** поле `blockId` — не отдельным полем.
3. **`regionFooter(vr.id)` рендерится один раз на вариант** (`:746`), независимо от `caption` — реплаи
   агента по `blockId===vr.id` теперь видны всегда.
4. **0 правок `scripts/server.py`.** Обе фичи укладываются в существующие поля item и существующий
   дедуп; синхронизирована только документация скилла (`skills/feature/dashboard-guide.md`,
   `feedback-loop.md`), т.к. меняется **смысл** контракта для агента.

## Почему

- **Реюз контракта дешевле и надёжнее новых полей/эндпоинтов.** Любое новое поле потребовало бы правки
  `_draft_add` (он собирает item из конкретных `body.get(...)`, а не из `body` целиком) и риска
  рассинхрона фронт↔сервер↔агент. `blockId`/`questionId`/`text` уже несут всю нужную семантику.
- **Дедуп по `questionId` бесплатно даёт взаимное исключение** «опция ↔ свой ответ»: не нужно ни флага,
  ни логики «какой ответ главнее» на сервере — последний записанный `answer` побеждает.
- **Свободный ответ без флага** держит модель простой: агент всегда читает `answer.text`; ему не нужно
  ветвиться на «свободный/выбранный», достаточно не предполагать, что текст ∈ `options`.
- **Единый `regionFooter` на якорь** убирает дублирование реплаев и чинит исходный баг «реплай к
  варианту без caption невидим» — один источник истины рендера на `blockId`.

## Последствия

- **Для оркестратора/агента (чтение submission):** `comment.blockId` теперь бывает ещё и
  `variants[].id`; `answer` по `questionId` может содержать **свободный текст вне `options`** —
  обрабатывать его как ответ человека, не отбрасывать как «невалидную опцию». Реплай в `replies.json`
  по тому же `blockId`/`questionId` рендерится под вопросом/вариантом. Контракт зафиксирован в
  `skills/feature/dashboard-guide.md` и `skills/feature/feedback-loop.md` (пример reply с `blockId:"v2"`).
- **Жёсткий набор полей item** остаётся ограничителем: будущие виды обратной связи либо влезают в
  существующие поля, либо требуют осознанной правки `_draft_add`.
- Несколько комментариев на один вариант допустимы (поле чистится после отправки) — все уходят в draft
  отдельными items.

## Альтернативы (отклонено)

- **Отдельное поле/флаг «свой ответ»** (напр. `freeform:true`) или отдельный `kind` — потребовало бы
  правки `_draft_add` и усложнило контракт без выгоды: `text` вне `options` уже однозначен.
- **Отдельное поле `variantId`/`targetId` для коммента к варианту** (+1 строка в `_draft_add`,
  `exploration-backend.md:99`) — отклонено в пользу реюза `blockId`: вариант и так имеет стабильный id,
  который уже служит якорем для реплаев.

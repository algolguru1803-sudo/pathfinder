# ADR-0009: Достоверная по-агентная атрибуция через транскрипты + ленивый `/trace/actions`

- **Дата:** 2026-06-13
- **Статус:** принято
- **Задача:** agent-trace-details
- **Отношение:** **поправка/дополнение к ADR-0003** (не отмена). Lane из хука остаётся best-effort.

## Контекст

Брифу `agent-trace-details` нужен по каждому агенту (под-агент и оркестратор) **понятный, достоверный**
список того, что он делал: тулзы, Bash, вызовы MCP (`mcp__*`), под-задачи — с ключевым аргументом и
статусом ok/running/error. Имеющаяся живая лента группирует действия по `lane`, но lane **best-effort**
(ADR-0003): исполнителя `tool.*` в payload хука нет, при нескольких параллельных под-агентах атрибуция
неточна. Дополнительно exploration вскрыл два факта, которых не было в базе знаний:

- рядом с под-агентским транскриптом CC пишет sidecar `agent-*.meta.json` = `{agentType, description,
  toolUseId}`, и его `toolUseId` **точно совпадает** со `spanId="span-"+toolUseId` из `subagent.start`;
- у одной задачи может быть **несколько** session_id (поправка к утверждению ADR-0003 «один на всех»).

Развилки: (а) тащить достоверную атрибуцию на горячий путь ленты `/trace/feed`; (б) утяжелять
`build_trace` чтением всех `tool_use` из транскриптов; (в) отдельный ленивый эндпоинт по образцу
`/trace/messages`. Плюс: ловить ли `mcp__*` в телеметрию и как (новое событие vs дозапись полей).

## Решение

1. **Точная по-агентная атрибуция — постфактум/лениво, через транскрипт + `meta.json.toolUseId`.** Мост
   «спан → транскрипт под-агента → все его `tool_use`» детерминирован (sidecar). Все `tool_use` в
   транскрипте принадлежат именно этому агенту → атрибуция точна, **в обход** ненадёжного lane.
2. **Новый ленивый `GET /trace/actions?slug&agent&session`** (`_trace_actions`, `scripts/server.py:394`;
   читалка `parse_transcript_actions`, `scripts/_aipf.py:451`) — читает **ОДИН** транскрипт по явному
   раскрытию карточки, read-only, mtime-кэш ~3 с per-(slug, agent). Контракт: `{description, actions:[{type,
   name, arg, status, ts, relMs}], counts, pending}`, действия отсортированы по `relMs`. `description` —
   из `meta.json` (`_agent_description`, `scripts/server.py:439`).
3. **MCP в ленте — дозаписью полей, той же `tool.*`-веткой.** `tool_name.startswith("mcp__")`
   (`scripts/telemetry_hook.py:134`) + новые поля `tool.start`: `kind` (`mcp`/`bash`/`tool`), для MCP
   `server`/`mcpTool` (`_parse_mcp_name`, `:195`). Старый формат/порядок `tool.*` не тронут.
4. **`summary` (описание задачи) пробрасывается и в `/trace`** (`_agent_record(..., summary=...)`); у
   оркестратора собственной задачи нет → авто-подпись `"оркестратор сессии"` (q5=A).
5. **Все по-агентные агрегаторы идут по ВСЕМ session_id задачи** (поправка к ADR-0003).

## Почему

- **Горячий путь ленты нельзя грузить транскриптами (ADR-0001).** `/trace/feed` читает только хвост
  `telemetry.jsonl` и не открывает прозу; чтение транскрипта на каждый тик дало бы задержку и сломало
  delta-only-модель. Поэтому достоверный список — ленивый, отдельным эндпоинтом.
- **Не утяжелять `build_trace`.** Он читает `telemetry.jsonl` целиком и под кэшем 3 с считает usage по
  `parse_transcript_usage` (только цифры). Подмешать туда чтение всех `tool_use` всех агентов — дороже и
  лишнее: actions нужны только по раскрытию конкретного агента. Отдельный per-agent эндпоинт читает один
  транскрипт по запросу — минимально инвазивно и согласовано с «не тащить лишнюю прозу».
- **`meta.json.toolUseId` — детерминированный ключ.** Он надёжнее, чем `attributionAgent` (тот пуст у
  `user`-записей) и чем lane-эвристика. Это обходит фундаментальное ограничение ADR-0003, не отменяя его:
  ограничение про payload хука остаётся в силе, мы просто берём истину из другого источника постфактум.
- **MCP дозаписью, а не новым событием `mcp.*`.** Та же `tool.*`-механика бесплатно даёт реалтайм,
  `running` и склейку по `spanId`; новые поля игнорируются Langfuse-форвардером (`_aipf.py`), курсор не
  ломается (инвариант «только дозапись»). Альтернатива (отдельный тип события) удвоила бы код без выгоды.

## Последствия

- **Две модели сосуществуют по дизайну:** realtime-лента (`/trace/feed`, lane best-effort, дёшево) +
  достоверная ленивая по-агентная сводка (`/trace/actions`, по транскрипту). Они не конкурируют: первая —
  «что происходит сейчас», вторая — «что агент сделал, точно».
- **Стоимость чтения транскрипта** (он бывает большим) гасится mtime-кэшем ~3 с per-(slug, agent) и тем,
  что читается только по явному раскрытию.
- **Мост надёжен в пределах одного прогона:** `toolUseId` дрейфует между прогонами — матчить по
  `toolUseId`, не по позиции (отражено в подводных камнях области телеметрии).
- **Поддерживать инвариант** «по-агентный агрегатор перебирает ВСЕ session_id задачи» — забыть про это
  значит потерять под-агентов дочерних сессий.
- ADR-0003 остаётся в силе для ленты; этот ADR — её дополнение для посмертного/ленивого пути.

# ADR-0012: Консенсус `/improve` = рой scout → vote-панель → детерминированная агрегация оркестратором

- **Дата:** 2026-06-13
- **Статус:** принято
- **Задача:** improve-workflow

## Контекст

Команда `/improve` должна обследовать существующее приложение **роем** аналитиков (каждый со своей
призмы) и **консенсусом** отобрать/ранжировать лучшие кандидаты-улучшения для гейта выбора фич. Бриф
прямо требует «консенсус» — но в плагине действуют два жёстких инварианта (`areas/orchestrator-skills.md`):

- **субагенты не спавнят субагентов** — Agent tool есть только у оркестратора, прямых каналов агент↔агент
  нет; все хэндоффы мёдиирует оркестратор;
- **`model` глобальна для `subagent_type`** — два режима одного агента идут на одной модели.

«Консенсус» наивно хочется реализовать как одного агента, который сам опросит рой и сведёт мнения. Это
**невозможно** (субагент не спавнит субагентов) и **нечестно** (один LLM, агрегирующий мнения, склонен к
самосогласованию и теряет независимость голосов). Прецедент уже был: панель `np-judge` в `/new-product`
(ADR-0006/0007) — независимые судьи + детерминированный `decision()` оркестратора.

## Решение

Консенсус собирается **оркестратором** из независимых вкладов одного двухрежимного агента `wf-improver`:

1. **Рой scout (фан-аут).** Оркестратор спавнит **7 `wf-improver` в scout-режиме параллельно**, по одной
   на призму (UX/продукт, производительность, надёжность, техдолг, DX, пробелы фич, доступность +
   безопасность — `skills/improve/phases.md:27`). Каждый читает `INDEX.md` первым и отдаёт кандидатов по
   схеме `### cand:` (`agents/wf-improver.md:38`). Сырьё → `scout/<prism>.md`.
2. **Консолидация + дедуп (оркестратор).** Все `scout/*.md` сливаются в один список с **дедупом** по
   «той же сути изменения» (по затронутым файлам/areas, не по формулировке): union `areas`, запоминание
   призм-источников (для tie-break), консервативный выбор `size`/`risk`/`impact`. Назначаются **стабильные**
   id `cand-1…cand-N`, пишется `candidates.md` (`skills/improve/consensus.md:31`).
3. **Vote-панель (фан-аут).** Оркестратор спавнит **3 `wf-improver` в vote-режиме параллельно**. Каждый
   видит **весь** консолидированный список и **независимо** оценивает каждый кандидат на шкале 0–3:
   `impact/effort/risk/confidence` + `verdict keep|drop` + `note` (`agents/wf-improver.md:60`). Голосующие
   между собой не общаются.
4. **Детерминированная агрегация (оркестратор, не LLM).** Оркестратор сам считает балл из чисел
   (`skills/improve/consensus.md:64`):

   ```
   imp  = mean(impact);  eff = mean(effort);  rsk = mean(risk);  conf = mean(confidence)
   keep = (число keep-голосов) / 3                  # «согласие» = доля keep
   raw   = imp − w_e·eff − w_r·rsk
   score = raw · (conf / 3)                          # уверенность — множитель 0..1
   ```

   Дефолты `w_e = w_r = 0.5` (effort и risk каждый снимают полбалла за единицу; веса — ручки под бриф).
   Кандидат с `keep == 0` (никто не оставил) **отбрасывается** до ранжирования. Сортировка по `score` ↓,
   tie-break по `keep`, затем `imp`, затем `−eff`; берётся **топ-K = 6–8** (`cand-K → feat-K`).

Агрегат на кандидата (`{candId, impact, effort, risk, confidence, keep, score}`) пишется в
`state.json.votes[]`; сырьё голосов остаётся в `votes/<n>.md`.

## Почему

- **Панель честнее одного агрегатора.** Три независимых голоса дают разброс, который виден в `keep`
  (согласие) и в усреднении; один LLM, «сводящий мнения», подменяет это собственным нарративом. Это та же
  причина, по которой `/new-product` держит судей **изолированными** (ADR-0007): независимость измерений —
  ценность, её нельзя растворить в одном проходе.
- **Агрегация — у оркестратора, потому что субагенты не спавнят субагентов.** Даже если бы захотелось
  «агента-сводчика», он не смог бы опросить рой — Agent tool только у оркестратора. Поэтому консенсус
  **структурно** обязан собираться оркестратором; это не вкусовое решение, а следствие инварианта.
- **Детерминированный балл = прозрачный и воспроизводимый ранкинг.** Формула из чисел исключает «LLM
  передумал» между прогонами, делает гейт не чёрным ящиком (балл/согласие/impact показываются человеку,
  `skills/improve/consensus.md:98`) и позволяет крутить веса под бриф («только низкий риск» → выше `w_r`).
  Симметрично `decision()` у `/new-product` (ADR-0007) — арифметику считает оркестратор, не агент.
- **Confidence — множитель, не слагаемое.** Высокий impact, в котором панель не уверена, **демпфируется**,
  а не разгоняется — `score = raw · conf/3`.
- **Один агент на два режима — следствие «model глобальна».** scout и vote живут в одном файле
  `wf-improver` (без `model:`); режим задаёт промпт. Две модели потребовали бы второго файла (как `np-*`
  у `/new-product`, ADR-0006) — здесь это не нужно.

## Последствия

- **Жёсткая схема выхода агента — контракт.** Оркестратор парсит `### cand:`/`### cand-K` детерминированно;
  «свободные» ответы агента ломают консолидацию/агрегацию. Поэтому схемы в `agents/wf-improver.md` (русский
  текст, но машинно-парсимый каркас) менять только синхронно с парсером в `consensus.md`.
- **Стабильность `cand-K` обязательна** на весь прогон: голоса и карточки гейта (`feat-K`) ключуются по
  этим id; перенумерация после голосования рассыпет привязку.
- **Дедуп — ручная работа оркестратора** (LLM-суждение «та же суть»), а не код. Слабый дедуп = дубли в
  гейте; агрессивный = потеря граней. Сигнал «несколько призм нашли одно» сохраняется как tie-break.
- **Числа призм/панели — ручки.** 7 scout / 3 vote / топ-K 6–8 — дефолты (`skills/improve/consensus.md:181`);
  в `AIPF_EVAL=1` они фиксируются для воспроизводимого прогона.
- **Расширение паттерна панели судей.** Третий потребитель «независимые оценщики + арифметика оркестратора»
  (после `np-judge`); если появятся новые голосующие роли — тот же каркас.

## Альтернативы (отклонено)

- **Один агент-«консенсусник», опрашивающий рой** — невозможен (субагент не спавнит субагентов) и нечестен
  (самосогласование одного LLM вместо независимых голосов).
- **LLM ранжирует список («reorder these»)** — недетерминированно, невоспроизводимо, непрозрачно для гейта.
  Числовая агрегация оркестратором даёт стабильный, объяснимый порядок.
- **Один scout без разбиения на призмы** — теряет широту: разные призмы видят разные классы проблем;
  разбиение + дедуп даёт и охват, и единый список.

# ADR-0013: Выбор фич `/improve` поверх существующего контракта (0 правок сервера/HTML)

- **Дата:** 2026-06-13
- **Статус:** принято
- **Задача:** improve-workflow

## Контекст

Единственный гейт `/improve` — **человек выбирает, какие из топ-K кандидатов делать**. Это **главное
взаимодействие** воркфлоу. Нужен способ показать топ-K, дать человеку отметить «делаем/пропускаем» по
каждой фиче, забрать выбор и уйти в DISPATCH.

Сервер (`scripts/server.py`) **агностичен к содержимому**: `/data` отдаёт `dashboard.json` как есть,
`/draft`/`/submit`/`/signal`/`/wait` не валидируют контент (зафиксировано в ADR-0008). Но в дашборде:

- **`questions[kind:"choice"]` — это N отдельных radio-пар**, multiSelect-виджета («чеклист с галочками»)
  физически **нет** в `templates/dashboard.html`;
- гейт-кнопка одна — `approve-plan` («Утвердить план»), зашита в HTML;
- `draft.json` **не** в `READABLE_FILES` сервера — выбор виден агенту только **после** Submit.

Развилка для главного гейта:
- **Вариант A** — 0 правок: реюз `questions[choice]` + `approve-plan` как есть.
- **Вариант B** — одна **аддитивная** правка `dashboard.html` (без сервера): новая ветка рендера
  `feature-pick` (чекбоксы + одна кнопка «Запустить выбранные» = submit+approve атомарно), реюз
  `saveAnswer`/`[data-answer]`.

## Решение

**Выбран Вариант A** — выбор фич целиком поверх существующего контракта, **0 правок `server.py` и
`dashboard.html`**. Контракт `feat-K` (`skills/improve/dashboard-guide.md:77`):

1. **Карточка + вопрос на фичу, общий `id = feat-K`.** Для каждого топ-K кандидата `K`:
   - `planBlocks[]` карточка `{id:"feat-K", title, body}` — markdown с призма/проблема/изменение/
     объём·риск·impact/затронутые файлы (контекст для человека);
   - `questions[]` `{id:"feat-K", text, kind:"choice", options:["Делаем","Пропускаем"]}` — бинарный выбор.
   Общий `feat-K` связывает карточку с вопросом и маршрутизирует комменты/реплаи
   (`repliesByBlock`/`repliesByQ`).
2. **Свободный ответ валиден (ADR-0008).** Помимо двух опций человек может ввести свою формулировку
   («делаем, но без X»). Это приходит как обычный `answer` того же `feat-K` с `text` вне `options` — читать
   как выбор **и** как уточнение к брифу фичи, не отбрасывать. На фичу — **один** `answer` (radio и свой
   ответ перебивают друг друга).
3. **Дефолт «нет ответа = Пропускаем».** Фича без `answer` не попадает в submission (`saveAnswer` игнорит
   пустой ввод). Отсутствие `feat-K` в submission трактуется как «Пропускаем» — человеку не нужно кликать
   каждый radio, только нужные. Это прописывается в `summary` человеку.
4. **Обязательный порядок Submit → Approve.** `draft.json` не READABLE, поэтому: сначала «**Отправить**»
   (фиксирует выбор в `submissions/<n>.json`), затем «**Утвердить план**» (`approve-plan`). Если
   `approve-plan` пришёл без свежего submission — читать нечего, переспросить про Submit. `approve-plan`
   здесь = «диспетчим выбранные»: оркестратор берёт последний submission, собирает `feat-K` с
   `text=="Делаем"` (или свободным «делаем…») → `state.json.selected[]` → DISPATCH.

## Почему

- **Культ «0 правок» (ADR-0008).** Гейт укладывается в существующие поля item и существующий дедуп
  `answer` по `questionId` — новых полей/эндпоинтов/флагов не нужно. Бэкенд остаётся content-agnostic;
  риск рассинхрона фронт↔сервер↔агент нулевой.
- **Реюз `feat-K` как единого якоря** даёт привязку карточки↔вопроса↔коммента↔реплая бесплатно — та же
  механика стабильных id, что у `planBlocks`/`questions` в `/feature`.
- **Дедуп `answer` по `questionId` бесплатно даёт «один выбор на фичу»** — radio и свой ответ
  взаимоисключаются без серверной логики (последний записанный побеждает).
- **Обратная совместимость и предсказуемость.** Тот же gate-сигнал `approve-plan`, что у
  `/feature`/`/new-product` — оркестратор интерпретирует его по стадии (здесь «диспетчим выбранные»). Ни
  одного нового пути в сервере, который мог бы сломать существующие команды.

## Развилка A vs B и почему A

- **B лучше по UX** главного гейта: честный чеклист с чекбоксами + одна кнопка снимает обе оговорки A —
  «длинный список дублирующих radio» и «два клика в строгом порядке Submit→Approve» (источник ошибки
  оператора). B при этом **тоже без правок сервера** (одна аддитивная ветка рендера в HTML).
- **Выбран A как стартовый контракт**, потому что: (1) он не трогает HTML вообще — полное соответствие
  культу «0 правок сервера/HTML» (ADR-0008); (2) обратносовместим и проверяем без изменения шаблона;
  (3) функционально корректен (визуально длинный список — но рабочий). **B оставлен как готовый
  аддитивный апгрейд** гейта (`skills/improve/dashboard-guide.md:109`) — одна строго аддитивная ветка
  рендера для `questions[kind:"feature-pick"]`, без `server.py`. Не реализовывать B, пока явно не
  попросят. Решение по A/B было вынесено человеку демкой (`mockups/select-v1.html` vs `select-v2.html`).

## Последствия

- **Оговорки A — это контракт, не баг.** «Два клика в строгом порядке» и «нет ответа = Пропускаем»
  обязаны быть прописаны человеку в `summary` (`skills/improve/phases.md:66`), иначе оператор ошибётся.
- **Свободный текст на гейте несёт двойную нагрузку** (выбор + уточнение брифа) — DISPATCH прокидывает
  `answer.text` в `brief.md` посеянной фичи (`skills/improve/consensus.md:144`).
- **Апгрейд до B локализован** — одна ветка рендера в `dashboard.html` + описание в `dashboard-guide.md`;
  сервер и контракт submission не меняются, поэтому переход A→B не ломает оркестратор.
- **Гейт `/improve` — третий потребитель** паттерна «обратная связь поверх существующего контракта»
  (после `/feature`-фидбэка и `/new-product`-гейтов).

## Альтернативы (отклонено)

- **Новый `kind`/эндпоинт под мультивыбор с правкой `server.py`** — нарушает «0 правок сервера», даёт
  рассинхрон; существующего `questions[choice]` достаточно.
- **multiSelect-виджет немедленно (Вариант B как старт)** — лучше по UX, но трогает HTML; отложен как
  чистый аддитивный апгрейд, чтобы стартовать с нулём правок.
